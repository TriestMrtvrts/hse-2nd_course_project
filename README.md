# hse-2nd_course_project

# Design-Doc: Чат-бот / RAG-система  

## 1. Контекст проекта  
### 1.1 Бизнес-задача  
- Предоставление студентам оперативных ответов на вопросы об учебном процессе
  на основе официальных документов образовательных организаций 
- Снижение нагрузки на сотрудников учебных офисов 
- Обеспечение единой точки доступа к нормативной и справочной информации (насколько это возможно)

### 1.2 Целевая аудитория и пользователи  
- Студенты вузов 
- Абитуриенты
- Сотрудники учебных офисов

### 1.3 Ограничения и допущения  
- Система не является официальным источником,
  справочный характер
- Ответы на основе загруженных документов
- Актуальность информации зависит от своевременного обновления базы знаний

## 2. Архитектура решения  
### 2.1 Общая схема  
-  Запрос через tg-bot 
-  Запрос -> embedding -> поиск релевантных фрагментов документов
-  Контекст передается в LLM для получения ответа

### 2.2 Хранилище знаний / документооборот  
-  Источник данных - PDF, DOCX, HTML с сайтов вузов
-  Документы конвертируются в markdown формат
-  Текст разбивается на чанки и в векторную бд

### 2.3 Retrieval + Generation  
-  retrieval - семантический поиск по эмбеддингам
-  векторная бд - SQLite + sqlite_vec
-  Generation - LLM + retrieved context

### 2.4 Интеграции и интерфейсы  
-  Telegram Bot API
-  насчет LLM надо будет уже смотреть по ходу дела

### 2.5 Инфраструктура и развертывание  
-  docker-контейнер
-  возможность запуска локально + on-premise развертка
-  не зависим от облачных сервисов

## 3. Данные и качество знаний  
### 3.1 Сбор и предобработка данных  
-  берем документы из открытых источников 
-  очистка, нормализация и приведение к markdown
-  автообновленние базы знаний при изменении файлов

### 3.2 Векторизация и индексирование  
-  векторизация -> multilingual sentence-transformer
-  индексирование при изменении в документах
-  чанки и параметры эмбеддингов подберем эмпирически

### 3.3 Метрики качества знаний  
-  Recall@k для retrieval
-  Ручная оценка
-  Логирование запросов без перс. данных

## 4. Модель и генерация  
### 4.1 Выбор LLM и промптинг  
-  instruction-tuned LLM (с поддержкой рус яз)
-  Промпт ограничивает модель ответами исключтельно на основе контекста
-  system prompt - роль помощника студента/сотрудника УО/абитурента

### 4.2 Контроль качества ответов  
-  ответы только при наличии подходящего контекста
-  при отсутствии данных сообщим юзеру
-  не гененрируем ответы с потолка

### 4.3 Обучение/дообучение  
-  дообучение пока что не планируется
-  качество через retrieval + prompt engineering
